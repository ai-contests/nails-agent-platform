❓ 为什么频繁出现"请求太频繁"

根本原因：@sillyl12324/xhs-mcp 在内部启动 Playwright Chromium，这是一个 headless 浏览器。XHS 服务器能通过 UserAgent、fingerprint、行为模式识别出自动化浏览器，触发频率限制。

表现：page.waitForFunction: Timeout 30000ms exceeded — XHS 返回了非 React 页面（限流落地页），__INITIAL_STATE__ 不存在，等待超时。

方案原理被限流风险状态
@sillyl12324/xhs-mcp
（Playwright Chromium）		 启动独立 headless 浏览器，JS 渲染后读 __INITIAL_STATE__ 高 — 特征明显 当前使用 / fallback
XHSCDPFetcher
（真实 Chrome CDP）		 附加到用户已登录的真实 Chrome，复用真实会话 极低 — 无法区分正常用户 已实现 / 需开启 Chrome
开启真实 Chrome 方法：open -a "Google Chrome" --args --remote-debugging-port=9222，然后登录 xiaohongshu.com。pipeline 会自动切换到 CDP 模式，source_status() 中 xhs_cdp: True。

② Top 8 信号为什么不到 10？

信号收集流程与过滤漏斗

直接回答：上次测试跑时手动把 xhs_detail_candidate_n=5（默认 15）和 detail_top_n=10 留在默认，但因为部分 detail 拉取超时被丢弃，最终只有 8 条通过。正常生产跑每轮应有 60-70 条候选，Top 10 完全可以达到。
📊 信号漏斗（实测数字）
搜索：7 关键词 × 10
= 70 条原始 feed
（镭射美甲 0 条，实得 60 条）
↓ 去重
去重后：~56 条候选
↓ 按互动分排序，取 top max(detail_top_n=10, detail_candidate_n=5) = 10
发 detail 请求：10 条
（测试时 candidate_n 设了 5，所以只发了 10）
↓ 1 条 "Note not found"，1 条超时 → 降级用 search-level 数据
最终输出：8 条
生产建议参数：
limit_per_kw=20 · detail_candidate_n=20 · detail_top_n=10
这样候选池 ~120 条，detail 请求 20 条，最终稳定输出 10 条。
⚙ 关键参数说明
参数默认值测试时用值含义
limit_per_kw1010每个关键词最多取多少条 feed 进入候选池
detail_candidate_n155从排序后的候选中最多考虑多少条发 detail 请求（扩大此值 = 更多 detail 机会）
detail_top_n1010最终目标输出条数，detail 成功数达到这个数就停止
xhs_use_llm_tagsTrueFalse是否调用 Qwen/VLM 补全 tag（False = 纯规则，速度快）
③ 9宫格图片检测与切割

文件：nails_agent/tools/fetchers/xhs_mcp_fetcher.py → _classify_image() / _split_grid9()

🔍 检测算法：_classify_image(path) → "grid9" | "wide_strip" | "normal"

XHS 9宫格合图是把 9 张单甲照拼成一张正方形图发布，无法靠宽高比区分（和普通单图宽高比相同）。检测逻辑分三层过滤：

步骤条件结果
1. 宽高比检测ratio > 2.5 or < 0.4→ wide_strip（丢弃）
2. 尺寸下限w < 600 or h < 600→ normal（太小不可能是9宫格）
3. 近正方形ratio 不在 0.85–1.18→ normal
4. 网格线检测列均值曲线有 ≥2 个谷值 AND 行均值曲线有 ≥2 个谷值→ grid9 ✓

谷值检测原理：把图片转灰度 → 对每列/每行取像素平均 → 卷积平滑 → 找连续低于中位数 8 个灰度单位的区域。9宫格的分割线（通常是白色或黑色细线）会在亮度曲线上形成明显波谷。

col_mean = arr.mean(axis=0)   # 每列平均亮度
row_mean = arr.mean(axis=1)   # 每行平均亮度
# 如果横竖各有 ≥2 个波谷 → 判定为9宫格
✂ 切割算法：_split_grid9(path) → 返回最清晰的 ≤2 张

切割为均匀 3×3 九宫格，对每个 cell 计算清晰度分数，返回最高分的 1-2 个（避免重复选同位置）。

# 均匀切割
cw, ch = w // 3, h // 3
for row in range(3):
    for col in range(3):
        box = (col*cw, row*ch, (col+1)*cw, (row+1)*ch)
        cell = img.crop(box)

# 清晰度：Laplacian 算子方差（越高越清晰）
kx = [[0,1,0],[1,-4,1],[0,1,0]]  # Laplacian kernel
lap_var = variance(abs(convolve(gray, kx)))

# 综合分 = lap_var * cell_area（大格子加权）
# 最终返回 top 1-2 个 cell
切割后的 cell 图保存为 {stem}_cell{row}_{col}.webp，替换原始合图参与后续的视觉特征提取和 tag 打标。
④ Tag 打标系统（三层递进）

文件：nails_agent/services/tag_enricher.py + nails_agent/services/tag_rules.py

🏷 Tag 字段结构
字段含义示例
style_tags款式/工艺风格法式、猫眼、渐变、波点、蝴蝶结
color_tags颜色裸色、蓝色、白色、多巴胺
material_tags材料/工艺手法亮片、猫眼粉、光疗、镭射
scene_tags使用场景约会、日常、夏日、通勤、婚礼
tag_source来源标注rules:title / rules:detail / rules:detail+llm:qwen / +vision:qwen-vl
tag_confidence置信度 0-10.4（rules纯规则）→ 0.75（LLM补全后）
🔄 三层打标流程
层 1：规则提取
tag_source = "rules:title"
tag_source = "rules:detail"
从 source_title、caption、topics 正则匹配预设关键词。
detail 数据比 search 数据多 caption + topics，所以 detail 后打的标签更多。
速度快 不需要 API
↓ 如果 tag_confidence < 阈值 且 use_llm_tags=True
层 2：Qwen 文本 LLM
QwenTagEnricher
tag_source += "+llm:qwen-xxx"
把 source_title + caption 批量发给 Qwen，让模型以 JSON 格式补全四类 tag。
支持批处理（一次调用处理多条），节省 API 成本。
需要 DASHSCOPE_API_KEY
↓ 如果图片已下载 且 tag 仍然不足 且 use_llm_tags=True
层 3：Qwen-VL 视觉 LLM
VisionTagEnricher
tag_source += "+vision:qwen-vl"
把下载好的图片 base64 编码发给 Qwen-VL 多模态模型，直接从图像推断款式/颜色。
这一层在文字信息不足时（如无 caption 的图片帖）最有价值。
需要 vision LLM API key 最慢，约 3-8s/张
should_call_llm() 判断条件：当任一 tag 字段（style/color/material/scene）为空 AND tag_confidence < 0.7 时触发 LLM 补全。避免对已有丰富标签的 detail 数据浪费 API。
🧹 Tag 清洗规则

所有来源的 tag 经过 clean_tag() 过滤：

  • 剥离 XHS 话题 markup：[xxx]
  • 去除"美甲"后缀/前缀（"猫眼美甲" → "猫眼"）
  • 丢弃泛词：美甲、好看、显白、推荐、教程、爆款、种草、百搭 等 ~35 个
  • 每字段上限：style ≤5、color ≤4、material ≤4、scene ≤4

📐 甲型分类：classify_nail_length(crop)

从 Roboflow 分割出的单甲 crop（RGBA 透明背景），取 alpha 通道 bbox 计算高宽比：

条件甲型输出
h/w > 1.8长甲nail_shape_tag = "长甲"
1.2 ≤ h/w ≤ 1.8中甲nail_shape_tag = "中甲"
h/w < 1.2短甲nail_shape_tag = "短甲"
优化（本次修复）：甲型分类和颜色提取共用同一个 Roboflow crop，不再二次调用 API。extract_nail_visual_features() 直接返回 nail_shape_tag 字段。
🎨 颜色特征提取：extract_nail_visual_features()
步骤方法输出字段
像素采样椭圆注意力 mask（偏向指甲板中央上部）+ 亮度/饱和度过滤—
主色提取OpenCV KMeans k=4，最多 12000 个采样点dominant_palette
颜色分类规则引擎（HSV + LAB + metallic hint）primary_color_family, primary_color_name
颜色温度Hue 范围规则color_temperature: warm/cool/neutral
亮度/饱和度HSV V/S 分段brightness_level, saturation_level
对比度调色盘 L* 最大差contrast_level: high/medium/low

前提：ROBOFLOW_API_KEY 环境变量已设置。未设置时 fallback 全图提取（无 crop，但颜色精度下降）。

✅ 规范说明

Mock 文件的定位：web/data/ 中的 mock 文件（trend_signals.json、metric_snapshots.json 等）是 UI 的最终兜底（仅在 pipeline 完全没有产出时显示），不是开发时的数据来源。

触发真实数据的完整流程：

curl http://localhost:18060/health                    # → {"ok":true}
curl http://localhost:18060/api/v1/login/status       # → is_logged_in: true

# 2. 运行 pipeline（真实模式）
uv run python -m nails_agent.runner

# 3. 输出写入
# web/output/trend_signals.json
# web/output/metric_snapshots.json
# web/output/images/latest/raw/*.webp

# 4. Streamlit UI 自动优先读 web/output/（data_loader.py）
如果 bridge 没有运行：cd /path/to/project && node scripts/xhs_rest_bridge.mjs --port 18060
🔁 Mock fallback 触发条件（仅 UI 层）
# web/data_loader.py
def _load(filename):
    out = OUTPUT_DIR / filename
    seed = DATA_DIR / filename   # mock 文件
    if out.exists():
        data = parse(out)
        if data:          # ← 非空才用 output
            return data
        # output 存在但为空 → fallback mock（pipeline 跑了但 0 结果）
    return parse(seed)   # ← 只有这里会用 mock

即：mock 数据只有在 web/output/ 文件为空（pipeline 失败/0 信号）时才会显示。正常情况下永远是真实数据。
